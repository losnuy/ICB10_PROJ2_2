# 네이버 API 종합 분석 대시보드 UX/UI 리서치 및 개선 가이드라인 보고서

본 보고서는 `naver-api-app` 대시보드 프로젝트의 기존 코드를 다각적으로 분석하여 정보 설계(Information Architecture), 컴포넌트 배치, 데이터 시각화의 효율성 및 사용자 경험(UX) 측면의 페인 포인트(Pain Points)를 진단하고, 이를 해결하기 위한 구체적인 디자인 가이드라인과 개선 계획을 제안합니다.

---

## 1. 개요 및 리서치 배경

### 1.1 분석 목적
본 대시보드는 네이버 오픈 API(통합검색 트렌드, 쇼핑인사이트, 쇼핑 검색, 블로그, 카페, 뉴스 등)를 연동하여 마케터, 상품 기획자, 트렌드 분석가에게 실시간 시장 동향 데이터를 다각도로 제공하는 것을 목표로 합니다.
다수의 서로 다른 API로부터 비동기식으로 데이터를 조회하고 풍부한 시각화를 지원하기 때문에, **사용자가 정보를 혼동하지 않고 직관적으로 탐색할 수 있는 정보 위계의 정립, 일관성 있는 시각 디자인 시스템 구축, 매끄러운 반응형 피드백 제어**가 서비스 성공의 핵심 요소입니다.

### 1.2 대상 범위
- **메인 및 네비게이션**: `src/app.py` (사이드바 입력 및 글로벌 설정, 페이지 전환)
- **종합 분석**: `src/pages/integrated_analysis.py` (5개 탭 연동을 통한 단일 키워드 복합 분석)
- **트렌드 분석**: `src/pages/trend.py` (검색 트렌드), `src/pages/shopping_trend.py` (쇼핑 트렌드)
- **개별 채널 분석**: `src/pages/shopping_search.py` (쇼핑 상품), `src/pages/blog.py` (블로그), `src/pages/cafe.py` (카페), `src/pages/news.py` (뉴스)

### 1.3 리서치 방법론
- **Heuristic Evaluation (휴리스틱 평가)**: 제이콥 닐슨의 10가지 사용성 원칙을 기준으로 하되, 데이터 시각화 및 Streamlit 프레임워크의 특성을 반영하여 아래 4대 진단 축을 설정합니다.
  1. **정보 설계 및 탐색 흐름 (Information Architecture)**
  2. **레이아웃 및 일관성 (Layout & Consistency)**
  3. **데이터 시각화 효율성 및 가독성 (Data Visualization)**
  4. **피드백 메커니즘 및 로딩 경험 (Feedback & Loading)**

## 2. 현 대시보드 UX/UI 진단 결과 (Pain Points)

### 2.1 정보 설계 및 탐색 흐름 (Information Architecture)
- **글로벌 설정과 로컬 설정의 역할 분담 모호**:
  - 사용자 관점에서 '분석 검색어(쉼표 구분)'와 '조회 기간'은 사이드바(`app.py`)에 글로벌 파라미터로 입력하도록 되어 있습니다.
  - 하지만 개별 페이지(예: `integrated_analysis.py`, `shopping_search.py`)로 진입하면 화면 상단에 '핵심 키워드 선택' 등의 셀렉트박스가 중복으로 노출됩니다. 글로벌 설정에서 입력한 키워드들이 어떻게 개별 페이지에 매핑되는지 시각적 힌트가 부족하며, 사용 흐름에 단절감을 줄 수 있습니다.
- **API 키 미입력 상태에서의 불명확한 유도**:
  - API 키가 `.env`나 `secrets`로 자동 로드되지 않았을 때, 사이드바와 메인 화면 전체에 경고 메시지(`st.warning`)와 안내 메시지(`st.info`, `st.stop()`)가 다발적으로 발생하여 시각적 혼란을 야기합니다.
  - 사용자가 첫 진입 시 무엇부터 해야 하는지에 대한 온보딩(Onboarding) 시각 흐름이 매끄럽지 못합니다.

### 2.2 레이아웃 및 일관성 (Layout & Consistency)
- **반응형 대응 한계 및 화면 찌그러짐 현상**:
  - `st.columns`를 이용해 화면을 2열 혹은 3열로 분할한 페이지가 많습니다. (예: `trend.py` 상단 필터 3열, `shopping_search.py` 통계 카드 4열, 차트 2열 등)
  - 데스크톱 모니터가 아닌 노트북이나 태블릿 등 가로 해상도가 좁은 디바이스에서 실행 시, `st.metric`의 값이나 Plotly 범례(Legend)가 잘리거나 세로로 심하게 왜곡되는 현상이 관찰됩니다.
- **비동기 탭 구조(`st.tabs`)의 연산 비효율성**:
  - `integrated_analysis.py`에서는 5개의 탭을 정의하고 있습니다. Streamlit의 작동 메커니즘상, 사용자가 특정 탭만 선택하더라도 탭 내부의 모든 코드가 순차 실행되어 5개의 API가 동시에 호출될 수 있습니다.
  - 이는 사용자가 원치 않는 데이터까지 수집하게 만들어 첫 페이지 로딩 속도를 매우 느리게 만들고 API 할당량 소모를 가속화하는 원인입니다.

### 2.3 데이터 시각화 효율성 및 가독성 (Data Visualization)
- **일관되지 않은 차트 컬러 테마 및 시각화 템플릿**:
  - 각 개별 분석 페이지마다 Plotly Express를 활용해 막대 그래프, 도넛 차트 등을 그리고 있으나 적용된 컬러 스케일이 제각각입니다.
    - 블로그 (`blog.py`): `Viridis` 컬러 스케일 사용
    - 카페 (`cafe.py`): `Cividis` 컬러 스케일 사용
    - 뉴스 (`news.py`): `Temps` 컬러 스케일 사용
  - 이로 인해 하나의 브랜드 서비스라는 일관성이 결여되고, 각 페이지로 이동할 때마다 시각적 톤앤매너가 단절되어 시각적 피로감을 줍니다.
- **텍스트 과다 채널의 시각적 요소 부재**:
  - 카페(`cafe.py`)와 뉴스(`news.py`) 탭 및 페이지는 수집된 웹 데이터가 텍스트 리스트 형태로 노출되는 비율이 매우 높습니다.
  - 단순 표(DataFrame) 위주의 나열은 사용자가 데이터의 핵심 트렌드나 이상 동향을 빠르게 훑어보는(Scannability) 속도를 현저히 떨어뜨립니다.
- **차트 선택의 중복 및 비효율성**:
  - `shopping_search.py`에서 상품의 가격 분포를 시각화하기 위해 `Box Plot`과 `Histogram`을 나란히 배치했습니다. 두 차트는 동일한 '가격 분포 범위와 빈도'라는 속성을 전달하므로 시각적 중복이 발생하며, 화면 공간을 비효율적으로 낭비하고 있습니다.

### 2.4 피드백 메커니즘 및 로딩 경험 (Feedback & Loading)
- **즉각적인 Rerun으로 인한 버벅임**:
  - 사이드바에서 날짜를 하루만 변경하거나, 텍스트 상자에서 키워드를 한 글자 수정하는 즉시 Streamlit 전체 페이지가 다시 로드되며 API 재요청이 발생합니다.
  - 잦은 네트워크 딜레이로 인해 입력 컨트롤러 반응이 느려지며, 불필요한 API 비용이 유발됩니다.
- **스피너 지연 및 진행 상태 표시 부족**:
  - 대용량 뉴스나 쇼핑 상품(최대 100개)을 수집하고 텍스트 형태소 분석을 진행하는 과정에서 단순 `st.spinner("로딩 중...")` 메시지만 표시됩니다.
  - 수집 데이터 정제 -> 텍스트 분석 -> 시각화로 이어지는 파이프라인의 진행 상황을 사용자가 인지할 수 없어, 시스템이 멈춘 것 같은 불안감을 심어줍니다.

## 3. UX/UI 개선 디자인 시스템 및 가이드라인

데이터의 가독성을 높이고 일관된 사용자 경험을 제공하기 위해 아래와 같이 UI 요소 표준화 및 비주얼 시스템 가이드라인을 제안합니다.

### 3.1 브랜드 컬러 시스템 (Color System)
네이버 API 대시보드로서의 브랜드 정체성과 모던한 비즈니스 분석 툴의 톤을 조화롭게 융합한 컬러 팔레트를 제안합니다.

- **주요 브랜드 컬러 (Brand Colors)**:
  - **Primary**: `Naver Forest Green (#03C75A)` - 네이버의 핵심 아이덴티티 컬러로 주요 강조 영역 및 긍정 메트릭에 적용.
  - **Secondary / Accent**: `Deep Teal (#0E7490)` - 신뢰감을 주는 비즈니스 데이터 분석 테마 컬러로 차트의 주 색상에 적용.
- **데이터 시각화 컬러셋 (Data Viz Colors)**:
  - **범주형(Categorical)**: 5개 이상의 키워드나 브랜드를 비교할 때, 명도와 색상차가 뚜렷하여 색맹 사용자도 쉽게 구별 가능한 순차 배색 적용 (`#03C75A`, `#0E7490`, `#6366F1`, `#EC4899`, `#F59E0B`).
  - **순차형(Sequential)**: 단어 빈도 분석(TF) 차트 등에는 동일 계열의 명도 변화만 주는 `Teals` 혹은 `Blues` 단일 계열 컬러 스케일을 통일해서 사용하여 시각적 노이즈 제거.

### 3.2 타이포그래피 및 레이아웃 위계 (Typography & Typography Hierarchy)
- **기본 서체 설정**: 산돌고딕, Pretendard, Noto Sans KR 등 가독성이 높은 고딕(Sans-serif) 계열 폰트 스택을 구성하여 텍스트 데이터의 가독성을 극대화합니다.
- **글꼴 크기 위계**:
  - **H1 (페이지 타이틀)**: `2.0 rem` (Bold, 좌측에 대표 이모지 고정 배치)
  - **H2 (섹션 타이틀)**: `1.5 rem` (Semi-bold, 구분선 배치 권장)
  - **H3 (서브 섹션/차트 제목)**: `1.1 rem` (Medium, Slate Gray)
  - **Body (본문 및 상세 내용)**: `0.95 rem` (Regular)
  - **Caption (설명/출처)**: `0.8 rem` (Light Gray, 이탤릭 배제)

### 3.3 컴포넌트 표준화 가이드라인 (Component Standards)
- **필터 제어 영역**:
  - 메인 화면 상단에 필터를 배치할 때는 반드시 **3열(column)을 초과하지 않도록** 설계하여 가로폭이 좁아지는 환경에 유연하게 대응합니다.
  - `st.multiselect` 컴포넌트는 사용자가 값을 선택하지 않았을 때의 동작(예: '비워둘 경우 전체 검색으로 간주합니다')을 안내하는 `help` 텍스트나 placeholder를 의무적으로 명시합니다.
- **인사이트 메트릭 카드 (`st.metric`)**:
  - 1행당 3개 배치를 표준으로 하고, 부득이 4개 배치를 해야 하는 경우 모바일 뷰에서 카드 크기가 뭉개지지 않도록 모바일 최적화 레이아웃 분기 코드를 적용해야 합니다.
  - 데이터의 변화율(Delta)을 표시할 때 상승/하락의 색상(초록/빨강)이 의미하는 바가 시장 상황의 긍정/부정적 해석과 어울리는지 확인합니다.
- **표 (DataFrame) 구성**:
  - 모든 표는 `use_container_width=True`를 설정하여 100% 가로 너비를 채우도록 유도합니다.
  - 본문 요약이나 링크처럼 길이가 긴 텍스트는 `column_config`를 적용하여 `width="large"` 또는 `LinkColumn` 형태로 포맷을 지정하여 텍스트가 겹치거나 잘리지 않도록 정밀하게 컬럼 폭을 예약합니다.

## 4. 페이지별 구체적 UX/UI 개선 시나리오

### 4.1 메인 엔트리포인트 (`src/app.py`)
- **개선 과제**: 사이드바에 집중된 입력 양식과 잦은 Rerun 제어
- **개선 시나리오**:
  1. **설정 위계 분리**: 사이드바에는 변경 주기가 낮은 '네이버 API 설정 (Client ID, Secret)'만 배치하고, 핵심 분석 파라미터는 메인 화면 또는 사이드바 상단에 카드 형태로 분리하여 배치합니다.
  2. **분석 실행 버튼(Run Button) 도입**: 검색어 입력창 하단에 `st.button("🔍 데이터 분석 실행")`을 도입하여, 사용자가 타이핑하는 동안 매번 즉시 Rerun되는 것을 방지합니다. 사용자가 키워드와 기간을 모두 설정한 후 버튼을 클릭해야 비로소 API를 호출하도록 설계 흐름을 바꿉니다.

---

### 4.2 통합 키워드 상세 분석 (`src/pages/integrated_analysis.py`)
- **개선 과제**: 느린 로딩 속도 극복 및 탭별 레이아웃 일관성 유지
- **개선 시나리오**:
  1. **조건부 지연 로딩 (Lazy Loading) 구현**: 5개 탭의 데이터를 한 번에 로드하지 않고, 사용자가 탭을 클릭하여 활성화하는 시점에 해당 탭의 API만 비동기 호출하도록 구현 방식을 수정합니다. (Streamlit의 `st.session_state`를 활용한 탭 상태 분기 로직 적용)
  2. **카페/뉴스 탭 시각화 보완**: 텍스트 데이터프레임만 있는 카페와 뉴스 탭에 새로운 가공 데이터 시각화를 추가합니다.
     - **뉴스 탭**: 수집된 뉴스 기사의 발행 언론사 도메인 분포 차트 추가(Pie Chart).
     - **카페 탭**: 글 등록 수 기준 상위 카페 랭킹 시각화(Bar Chart).
  3. **통일된 2단 레이아웃**: 모든 탭의 내부 구성을 [좌측: 시각화 차트 / 우측: 요약 지표 및 데이터프레임] 형태로 동일하게 디자인하여 화면 전환 시 어색함을 해소합니다.

---

### 4.3 통합 & 쇼핑 트렌드 분석 (`src/pages/trend.py`, `src/pages/shopping_trend.py`)
- **개선 과제**: 다중 키워드 시각화 시 컬러의 가독성 및 필터 혼동 방지
- **개선 시나리오**:
  1. **키워드 컬러 매핑의 통일**: 사용자가 입력한 키워드들이 라인 차트와 하단 KPI 카드, 표에서 항상 동일한 고유 색상으로 매핑되도록 Plotly의 `color_discrete_map`을 명시적으로 선언합니다. (예: '캠핑'은 언제나 Teal, '글램핑'은 Indigo로 표시되어 사용자가 차트를 읽을 때 인지적 혼란을 없앰)
  2. **인사이트 카드의 반응형 배치**: 키워드가 4개 이상으로 많아질 경우 가로로 배치되는 `st.columns` 대신 2x2 또는 리스트 형태로 자동 개행되도록 하거나, 상위 3개 키워드만 핵심 메트릭으로 요약하여 보여주는 스마트 요약 구조를 차용합니다.

---

### 4.4 쇼핑 상품 분석 (`src/pages/shopping_search.py`)
- **개선 과제**: 가격 분포 차트의 중복 해결 및 이상치 해석 강화
- **개선 시나리오**:
  1. **차트 단일화 및 고도화**: 중복되는 Box Plot하고 Histogram 대신, **Violin Plot**을 도입하거나 **Box Plot 단일 배치**로 공간을 확보합니다.
  2. **이상치(Outlier) 시각적 피드백**: IQR(사분위수) 분석을 통해 탐지된 극단값(예: 시장가 대비 지나치게 비싸거나 싼 상품군)을 단순히 텍스트 설명으로만 제공하지 않고, 목록 내 데이터프레임에서 배경색이나 이모지 경고 아이콘(⚠️)으로 하이라이트 표시하여 사용자가 이상 상품을 즉시 분별할 수 있도록 돕습니다.
  3. **브랜드 점유율과 판매처 분포의 재배치**: 좌측에 브랜드 도넛 차트를 배치하고, 우측에 판매처 분포 가로 막대 그래프를 배치하여 1:1 대칭 구조의 완성도 높은 비주얼 대칭을 형성합니다.

## 5. 성능 및 기술적 개선 방향 (UX 최적화)

사용자 경험은 화면에 표시되는 속도 및 예외 상황에 대처하는 신뢰성과 직결됩니다. 이를 위한 백엔드/프런트엔드 연동 최적화 가이드라인입니다.

### 5.1 캐싱 전략 고도화
- **정밀한 캐시 만료 정책**: `st.cache_data` 사용 시 API별 특성에 맞게 TTL(Time-To-Live)을 설정합니다.
  - 검색어 트렌드 데이터(하루 단위 업데이트): `ttl=86400` (24시간 캐싱)
  - 실시간 뉴스 및 블로그 리뷰(시시각각 변동): `ttl=1800` (30분 캐싱)
- **사용자 키(Key) 최적화**: API ID/Secret 정보가 세션 내에서 변경되는 즉시 캐시가 유효하게 갱신되도록, 캐시 함수 데코레이터 인자에 `client_id`와 `client_secret`을 정확히 매핑하여 캐싱 오작동을 방지합니다.

### 5.2 예외 상태 (Error States) & 빈 데이터 상태 (Empty States) 설계
- **Empty States**: 검색 결과가 없거나 데이터가 비어 있을 때 화면이 하얗게 방치되지 않도록 고유 일러스트레이션이나 텍스트 템플릿을 제공합니다.
  - *예시*: "🔍 입력하신 키워드와 조건에 부합하는 상품 데이터가 없습니다. 검색어를 변경하거나 조회 수집 개수를 늘려보세요."
- **에러 핸들링**: API 오류가 발생하면 시스템 오류 메시지(예: `KeyError`, `HTTPError` 스택 트레이스)를 그대로 화면에 출력하지 않고, `NaverApiClient` 단에서 한글로 정제된 에러 원인과 대처 프로세스를 카드 컴포넌트로 깔끔하게 제공합니다.

### 5.3 데이터 처리 공통 유틸 모듈화
- 각 페이지에 개별 구현된 HTML 태그 제거(`clean_html`) 및 형태소 분석/불용어 필터링(`stopwords`) 로직을 `src/utils.py`와 같은 단일 모듈로 이관합니다.
- 공통 모듈화 시 페이지 간 로직 불일치 현상이 방지되고 코드 중복이 80% 이상 제거되어 화면 로딩 시 오버헤드를 크게 단축할 수 있습니다.
